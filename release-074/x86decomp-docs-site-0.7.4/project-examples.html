The hybrid model

A hybrid project keeps the program continuously buildable while each function advances independently through two validation lanes:

                         ┌─ Matching lane ─ byte/instruction reproduction
original bytes → source ─┤
                         └─ Functional lane ─ behavior reproduction

A function can pass functional validation without matching the original machine code, or byte-match without completing broader integration testing. The statuses are deliberately independent.

Matching decompilation

Measures whether the candidate reproduces the target’s declared machine-code properties.

not_started → decompiled → compiles → abi_compatible → instruction_similar → byte_matched → image_integrated → full_relink_validated

Functional decompilation

Measures whether the candidate reproduces behavior within declared finite models and scenarios.

not_started → decompiled → compiles → abi_compatible → differentially_validated → symbolically_bounded → integration_validated

Assume this example reconstructs:

target.exe
└── function pe-rva:00401230
1

Capture the target evidence

Start with the executable and any legally available supporting artifacts:

x86decomp target-pack-infer target.exe target-pack/ \
  --pdb target.pdb \
  --map target.map \
  --decisions target-decisions.json

x86decomp target-pack-verify target-pack/

The target pack records observed facts separately from operator decisions. Missing compiler, linker, language, or layout details remain unknown; the toolkit does not invent them.

Evidence rule. A decision can establish a project assumption or known fact only when the operator explicitly supplies it. Omitted facts remain unknown or become blockers.
2

Create a hybrid project

x86decomp project-from-target target-pack/ project/
x86decomp project-check project/

The generated project contains source lanes, validation policy, reports, durable pipeline definitions, blockers, project state, and immutable artifact storage. A target pack is the grounded automatic-template input; project creation must not emit fake source, fake toolchains, or fake link order.

Conceptually, the project begins like this:

project/
├── original/
│   └── target.exe
├── target-pack/
├── functions/
├── src/
│   ├── asm/
│   ├── staging/
│   └── accepted/
├── build/
├── reports/
├── orchestration/
└── state/
3

Export and import function analysis

Run Ghidra static analysis and export the resulting function packets:

x86decomp ghidra-export <ghidra-project> <program>

For each function, the exported material can include:

functions/pe-rva_00401230/
├── function.json
├── decompiler.c
├── listing.asm
├── raw-pcode.jsonl
├── high-pcode.jsonl
└── references.json

These are analysis observations, not authenticated original source. Ghidra output can contain incorrect boundaries, signatures, types, or control-flow interpretations, so downstream compilation and validation remain mandatory.

4

Generate the continuously buildable baseline

x86decomp hybrid-generate project/hybrid-project.json

Initially, every function is represented by assembly containing preserved bytes. The generator also creates staging C files and build metadata. This gives the project a buildable fallback before any function has been successfully reconstructed.

src/asm/sub_00401230.S       ← safe byte-form fallback
src/staging/sub_00401230.c    ← decompiler-derived candidate

Source maturation path

original bytes
    ↓
generated assembly
    ↓
decompiler-derived C/C++
    ↓
human-corrected C/C++
    ↓
accepted candidate

Unfinished or rejected functions stay on the assembly/original-byte path.

5

Build the baseline

For example:

cd project
make

The assembly fallback for sub_00401230 becomes a COFF object:

src/asm/sub_00401230.S
          ↓ assembler
build/sub_00401230.o

At this point the project is buildable, but the function has not yet been decompiled or validated.

function: pe-rva:00401230

matching:
  state: not_started

functional:
  state: not_started

active_source:
  stage: generated_assembly
6

Prepare and compile a source candidate

An analyst edits the staging output into a candidate such as:

int __cdecl calculate_total(int value, int adjustment)
{
    return value + adjustment;
}

The candidate is compiled with a registered, provenance-recorded compiler profile:

x86decomp compile compiler-profile.json \
  project/src/staging/sub_00401230.c \
  project/build/sub_00401230_candidate.obj \
  --report project/reports/compile/sub_00401230.json

A successful compile advances both lanes only as far as their independently justified state:

matching:
  state: compiles

functional:
  state: compiles
Compilation is not validation. A successful compiler invocation says nothing by itself about byte identity, ABI correctness, behavioral equivalence, or whole-image integration.

Check the ABI

Verify the calling convention, argument locations, stack cleanup, return registers, preserved registers, and relevant structure layout.

Suppose the evidence supports:

Architecture:       x86
Calling convention: __cdecl
Arguments:          stack words 1 and 2
Return value:       EAX
Stack cleanup:      caller

After a passing ABI report:

matching:
  state: abi_compatible

functional:
  state: abi_compatible

This is the last shared-looking milestone. From here, the lanes use different validators.

Lane A

Matching lane

Compare candidate code with the original function

x86decomp diff-function \
  project/original/target.exe \
  0x00401230 \
  16 \
  project/build/sub_00401230_candidate.obj \
  sub_00401230

The comparison examines the original PE range against the candidate object symbol. Depending on the result, the function may reach:

abi_compatible
      ↓
instruction_similar
      ↓
byte_matched

For example:

matching:
  state: instruction_similar
  reports:
    diff: reports/diff/sub_00401230.json

This means the normalized instructions or control flow met the declared comparison policy, but the emitted bytes were not yet identical.

After adjusting source, compiler flags, compiler profile, or known relocation handling, a later candidate might produce:

matching:
  state: byte_matched
Narrow claim. A byte match is a machine-code reproduction claim for the declared function range and normalization policy. It is not by itself proof of whole-program correctness.

Integrate the matched candidate

Once accepted, the project changes that function’s active source:

Before:
sub_00401230 → src/asm/sub_00401230.S

After:
sub_00401230 → src/accepted/sub_00401230.c

All other unfinished functions continue using assembly fallback objects.

Build the hybrid image:

x86decomp relink project/relink-manifest.json

Then compare the produced image under the target-specific profile:

x86decomp image-match \
  project/original/target.exe \
  project/build/target-hybrid.exe \
  project/image-profile.json

A successful per-function insertion advances matching to:

byte_matched → image_integrated

A successful declared full-image relink gate can advance it to:

image_integrated → full_relink_validated
Lane B

Functional lane

Run differential tests

The functional lane compares behavior over explicitly declared inputs and observations:

x86decomp dynamic-validate \
  original-function.bin \
  candidate-function.bin \
  harness.json

A passing result advances:

abi_compatible → differentially_validated

This means the target and candidate behaved the same for that finite harness. It is not an all-input proof.

Run bounded symbolic validation

x86decomp symbolic-validate \
  original-function.bin \
  candidate-function.bin \
  --architecture x86 \
  --stack-argument-words 2 \
  --output-register eax \
  --max-steps 64 \
  --max-paths 16

A passing result advances:

differentially_validated → symbolically_bounded

The result applies only to the declared instruction, path, memory, alias, and observation bounds.

Run integration scenarios

Execute native application scenarios only in an explicitly approved isolation boundary:

x86decomp integration-run \
  project/integration/calculate-total.json \
  --external-wrapper project/scripts/run-in-vm

The scenario might compare:

exit code
stdout/stderr
generated files
selected process state
declared application behavior

A passing integration report advances:

symbolically_bounded → integration_validated
Execution boundary. Unknown native programs must not be executed directly on an ordinary development host. Integration execution belongs behind explicit consent and an appropriate VM, external sandbox, or reviewed wrapper.

Final state of the example function

The function can legitimately finish with different results in each lane:

function: pe-rva:00401230

active_source:
  stage: accepted_source
  path: src/accepted/sub_00401230.c

matching:
  state: instruction_similar
  reports:
    compiler: reports/compile/sub_00401230.json
    abi: reports/abi/sub_00401230.json
    diff: reports/diff/sub_00401230.json

functional:
  state: integration_validated
  reports:
    differential: reports/dynamic/sub_00401230.json
    symbolic: reports/symbolic/sub_00401230.json
    integration: reports/integration/sub_00401230.json

Exact interpretation

The reconstructed function passed the declared functional tests, bounded symbolic model, and integration scenarios, but it does not yet reproduce the original instruction sequence or bytes.

That is a valid hybrid-project outcome. The toolkit must not collapse it into a vague complete status.

Whole-project flow

Target executable and supporting evidence
                    │
                    ▼
             Verified target pack
                    │
                    ▼
              Hybrid project
                    │
                    ▼
       Ghidra function-artifact import
                    │
                    ▼
      Assembly fallback for every function
                    │
                    ▼
          Continuously buildable image
                    │
          ┌─────────┴──────────┐
          ▼                    ▼
   Matching validation   Functional validation
   compile               compile
   ABI                   ABI
   instruction diff      differential execution
   byte comparison       bounded symbolic check
   image integration     integration scenarios
   full relink
          └─────────┬──────────┘
                    ▼
       Evidence-backed project status
                    │
                    ▼
       Reproducibility and release gate

The important property is that replacing one function does not require finishing every function. Validated candidates move forward, while unresolved functions retain their byte-safe assembly fallback.
